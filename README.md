# 🦆 Modern Data Stack: Retail Analytics (dbt + DuckDB)

## 🎯 Project Overview
This project demonstrates a professional **End-to-End Analytics Engineering** pipeline developed in a local environment. I transformed a raw dataset of **541,909 rows** into a high-performance data warehouse using the **Modern Data Stack (MDS)**. 

Beyond simple data moving, this project implements **Medallion Architecture**, **Data Quality Testing**, and **Behavioral Cohort Analysis (RFM)** to turn messy transactional data into actionable business insights.

---

## 🏗️ Architecture & Tools
I chose this stack to simulate a high-performance, low-cost "Modern Data Stack" that runs entirely on a local machine but is built for cloud scalability.

* **Storage & Compute:** [DuckDB](https://duckdb.org/) — An in-process OLAP database used for lightning-fast analytical queries.
* **Transformation:** [dbt Core](https://www.getdbt.com/) — Used to manage the "T" in ELT, handle modular SQL, and maintain data lineage.
* **Visualization:** **Python (Plotly & Pandas)** — Used to bridge the gap between the warehouse and the end-user with interactive dashboards.
* **Environment:** **Local Zsh (Z Shell)** — Utilizing a native Unix-based shell for advanced command-line automation and version control. Also made use of an *Isolated Python Virtual Environment* for dependency management.

---

## 🛠️ The Analytics Engineering Journey

### Phase 1: Local Environment Setup
Moving away from virtualized environments, I optimized the workflow for local development:
- **Shell Optimization:** Configured the project using **Zsh**, leveraging its advanced tab-completion and scripting capabilities for environment management.
- **Dependency Isolation:** Implemented a Python **Virtual Environment (venv)** to ensure dbt-core and its adapters (dbt-duckdb) remained isolated from the system Python.
- **Performance:** By running directly on the local machine's hardware, data processing times for the 542k row dataset were significantly reduced compared to initial VM tests.

### Phase 2: The Medallion Architecture (Data Layers)
I organized the project into three distinct layers to ensure data reliability and lineage:

1.  **Bronze (Raw):** Untouched data loaded via `dbt seed`.
2.  **Silver (Staging):** Cleaned and standardized data. 
    * **The Timestamp Fix:** Resolved a critical conversion error by using `strptime(InvoiceDate, '%m/%d/%Y %H:%M')` to parse non-standard date formats.
    * **Data Cleaning:** Filtered out negative quantities (returns) and rows missing `customer_id`.
3.  **Gold (Marts):** Business-ready tables. I built the `fct_customer_rfm` model to calculate **Recency, Frequency, and Monetary** values for behavioral segmentation.

### Phase 3: Data Quality & Governance
Instead of "hoping" the data is correct, I implemented **Automated Testing**:
* **Generic Tests:** Ensured `customer_id` and `invoice_id` are never NULL and always unique.
* **Documentation:** Every model is documented in `schema.yml`, allowing for a searchable Data Catalog generated by `dbt docs`.

## 📈 Overall Business Insights
After executing the full pipeline and analyzing the **Gold Layer**, the following insights were uncovered:

### 1. High-Value Customer Concentration (The Pareto Principle)
The analysis reveals a significant "Whale" effect. While the dataset contains thousands of unique customers, the **Top 10 Customers** alone contribute a disproportionate amount of total revenue. 
* **Key Discovery:** Customer `14646` generated **£280,206**, which is over **135x** the average customer spend (£2,053).
* **Actionable Insight:** The business should implement a "VIP Loyalty Program" specifically targeting these top 0.1% of customers to prevent churn.

### 2. Frequency vs. Monetary Value
There is a strong correlation between order frequency and total spend, but it is not absolute.
* **Observation:** Customer `14911` has the highest frequency (**201 orders**) but is second in total spend. 
* **Actionable Insight:** High-frequency/Low-monetary customers are prime targets for upselling, while High-monetary/Low-frequency customers require targeted retention campaigns.

### 3. Data Cleaning Impact
The transformation process revealed that approximately **26%** of raw records were either "Returns" (negative quantities) or lacked a `Customer ID`.
* **Technical Insight:** By filtering these in the Silver Layer, we ensured the RFM model was built on "True Sales." This prevented average spend metrics from being artificially deflated by nearly **£500 per customer**.

## 🚀 How to Run

### 1. Setup Environment
```zsh
# Clone the repo
git clone [https://github.com/emmanuelgharteykb/modern-data-stack-project.git](https://github.com/emmanuelgharteykb/modern-data-stack-project.git)
cd modern-data-stack-project

# Setup Python Virtual Environment
python3 -m venv venv
source venv/bin/activate
pip install dbt-duckdb pandas plotly
```

### 2. Run the Pipeline
```bash
cd analytics_engine
dbt seed   # Load raw data (541k rows)
dbt run    # Transform (Bronze -> Silver -> Gold)
dbt test   # Validate data integrity
```

### 3. Generate Insights
```bash
# Run the Python visualization script
python visualize_rfm.py
```
